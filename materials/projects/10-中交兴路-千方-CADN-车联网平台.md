# CADN 车联网分布式接入与消息分发平台

**公司**: 中交兴路（CTFO）/ 千方集团 | **时间**: 2009.10–2011.05；2012.04–2015.05 | **角色**: 开发工程师 → 开发经理 / 高级软件工程师

**技术栈**: C++ · epoll · TCP 长连接 · GB/T 808 · Redis · Oracle · 分布式消息总线 · 一致性哈希 · 服务发现

---

## 项目背景

CADN v6.4 是面向车联网（GB/T 808 终端）的分布式接入与消息分发平台，支撑交通部全国车辆监管平台及中交兴路车联网业务。项目始于 2015 年前后，在无 Kubernetes、Kafka、gRPC 的年代，采用 C++ 多进程 + TCP 长连接 + 私有二进制协议，自研了一套具备微服务特征的分布式系统。

## 架构设计

系统采用**多进程微服务架构**，按功能纵向拆分为四层：

```
终端层 (GB/T 808)
    ↓ TCP
接入层: was/ — 终端长连接接入、鉴权、黑白名单
    ↓ 原始 808 字节流
协议层: scmsg/ — 808 解析、分包、转内部协议
    ↓ 内部 commproto 包
业务层: msg/ — 登录、订阅、匹配、分发
    ↓ 需持久化/查询
数据层: datapool/ → Oracle / MongoDB / Redis
```

**核心设计亮点：**

- **分层接入**：`was/` 只负责维持海量终端 TCP 长连接，原始字节流转发给后端 `scmsg/` 或 `data_proxy/`，接入层可独立水平扩展，协议升级不影响连接管理。
- **协议中立**：`GBHandler` 完成 808 解包/组包/校验和/粘包拆包，`InterProtoConvert` 统一转换为内部 `commproto` 二进制协议，兼容多 OEM 版本扩展。
- **分布式消息总线**：`msg/` 作为系统 Pub/Sub 中枢，业务方按车辆 ID / 命令类型 / 省域 / 用户组订阅；`JobDispatch` 线程池异步化处理，避免单消息阻塞后续流量。
- **无共享状态**：服务间不共享内存，状态通过 Redis 和 TCP 消息同步；`addrsrv/` 收集节点负载，`datacenter` 广播服务列表变更实现简单服务发现。
- **数据一致性同步**：`datacenter/synmgr` 定时从 Oracle RAC 视图拉取车辆/企业/终端绑定关系增量，写入 Redis 并广播 `MOD` 消息通知各节点刷新本地缓存，最终一致性模型支撑高并发读取。

## 技术决策

- **为什么自研 epoll 异步网络库**：libevent 的 C 接口在 C++ 项目中使用不便（回调无法绑定成员方法），自研库利用 C++11 lambda 和 std::function 实现类型安全的异步回调。
- **为什么一致性哈希而不是取模**：车机设备频繁上下线会导致集群节点变化，取模哈希会导致大量缓存失效。一致性哈希将影响范围控制在相邻节点，缓存命中率更高。
- **为什么不用消息队列**：2015 年 Kafka 尚未普及到团队技术栈，且车联网对延迟极度敏感（心跳、报警、指令下发需毫秒级响应），自建 TCP 长连接总线可精确控制序列化、重连、心跳、超时。

## 性能数据

- 系统集群化程度从 30% 提升至 80%
- 车机上线异常从 1 例/周降至 <1 例/3 月（数据一致性方案效果）
- 单进程并发从 5w 提升至 25w（锁优化），消息转发能力 7w/s
- 在线用户哈希桶 65535 个，接入层按十万级并发设计
- 20+ 独立微服务进程，通过统一 `commproto.h` 二进制协议通信

## 踩坑与解决

- **集群数据一致性**：多节点之间的设备状态同步延迟导致车机上线失败。通过静态数据广播（配置类）+ 动态数据扩散（状态类）+ 多级缓存管理（热点数据）三策略组合解决。
- **锁竞争瓶颈**：早期使用全局互斥锁，高并发下锁竞争成为瓶颈。通过锁粒度拆分（per-connection mutex）+ 无锁队列解决。
- **GPS 协议复杂度**：808/809 国标协议是二进制协议，字段众多（100+），含嵌套结构体和变长字段。通过协议状态机 + 位域解析器实现高效解析。
- **稳定性考验**：对接广州亚运会"两客一危"车辆 GPS 监控系统，国家级项目不允许出错，多次赴实地运维积累生产环境排障经验。
