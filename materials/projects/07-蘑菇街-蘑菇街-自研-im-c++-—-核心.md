## 架构设计

长连接 SDK：基于 epoll（Linux）/kqueue（macOS）的异步非阻塞 I/O 引擎，提供分包组包（自定义二进制协议）、
状态机（连接→鉴权→就绪→断开）、自动重连（指数退避）、加密（TLS + 自定义加密层）、网络诊断（心跳+超时检测）。
IM2.0 服务端：接入服务器（长连接管理 + 消息路由）、登录鉴权（Token/OAuth）、会话管理（多端同步 + 消息队列）、
AppServer（业务逻辑 + 推送）。

## 技术决策

- 为什么自研而不直接用 XMPP/MQTT：业务需要强定制（如消息撤回、多端同步语义、自定义加密），标准协议的表层扩展无法满足。自研协议可以精确控制包格式和握手流程，减少冗余开销。
- 为什么跨平台 SDK：公司同时有 iOS/Android/Web/PC 四个客户端，统一 SDK 避免多端重复开发和协议不一致。

## 性能数据

SDK 推广为全公司通用长连接基础组件，降低多端接入成本。
IM2.0 支撑亿级消息规模，高并发下长连接稳定性 >99.9%。

## 踩坑与解决

- 向前兼容：IM1.0 协议正在服役，IM2.0 需要在同一端口上同时支持新旧协议。通过协议版本号协商 + 握手阶段判断版本 + 双协议栈共存过渡解决。
- 无法灰度：长连接服务无法像 HTTP 服务一样通过 LB 按比例分流。通过客户端分批升级（版本号策略） + 服务端双栈兼容实现平滑迁移。
