# Phase 1 — 当前 `data/resume.yaml` Information Architecture 分析

## 分析依据

- `PRD.md` v3.0
- `materials/facts/facts.yaml`
- `CLAUDE.md` 内容规则
- `materials/coarse/` 粗料

---

## 一、高层定位问题

### 1.1 当前简历没有区分 Resume Branch

当前 `resume.yaml` 是一份「混合版」简历，同时想表达 Security 和 Backend，结果两边都不够聚焦。

**问题表现**：
- Summary 中同时出现「Tcp/Http网关设计」「电商风控、甲方安全、乙方安全」「IM」「高性能系统」。
- 招聘经理在 20–30 秒内无法判断候选人最适合哪个岗位。
- 没有为 Resume A（Security）和 Resume B（Backend）分别建立清晰的职业身份。

**PRD 对照**：PRD 第 2 章要求简历降低招聘风险，强化专业身份。当前简历让招聘经理的风险判断成本变高。

### 1.2 Summary 开篇是个人特质，不是专业身份

当前 Summary：
> 热爱技术, 积极乐观, 自我驱动力强, 乐于挑战新事物；10+年Linux Server开发经验……

**问题**：
- 「热爱技术、积极乐观」是低信号词，对 Senior 岗位没有区分度。
- 前 10 秒阅读时间浪费在软技能上，没有建立「我是谁、我擅长什么」的认知。
- 没有直接对应目标岗位（Application Security Engineer / Senior Backend Engineer）。

**PRD 对照**：PRD 第 3.2 章「Maximize curiosity, not explanation」和第 3.3 章「Every sentence should create a question」。当前 Summary 没有引发好问题，只是在陈述。

## 二、Profile 区域问题

### 2.1 `birth_year` 不应出现

`birth_year: 1985.8` 与 `CLAUDE.md` 中「无出生年份」规则冲突。

### 2.2 `tech: C++/Rust/Go` 位置和价值不足

- 当前放在 `profile.tech`，但没有说明这三者的角色。
- 对 Security Resume 来说，这三门语言只是工具；对 Backend Resume 来说，应该更突出 Rust/Go。
- 建议将技术栈放入 Summary 或 Core Skills，而不是作为 profile 字段简单罗列。

### 2.3 缺少明确的目标岗位声明

当前 Summary 末尾：
> 求职意向： 资深开发工程师，架构师。

**问题**：
- 过于宽泛，没有区分 Resume A/B。
- 「架构师」与 PRD 中「不要 Principal Engineer、技术总监、纯架构师」的定位冲突。

---

## 三、Core Skills 问题

### 3.1 分类方式不适合两个 Resume Branch

当前 Core Skills 分为：编程语言、基础设施、工程域、业务域、工具与效率。

**问题**：
- 这是「技能清单」，不是「能力叙事」。
- 没有按 Resume Branch 区分优先级。
- 「工具与效率」中的 Claude Code / windsurf / VIM / Git / CET-6 对 Senior 岗位价值极低，占用预算。

### 3.2 缺少 ATS 关键词密度设计

PRD 第 17 章要求关键词在 Summary、Core Skills、最近项目中均匀分布。当前 Core Skills 没有明确对应 Resume A/B 的 ATS 关键词矩阵。

### 3.3 业务域描述偏黑话

当前：
> 安全领域：兼具甲乙方安全防护经验，多款安全产品的自研经验，具备复杂场景下的黑灰产对抗实战经验。

**问题**：
- 「黑灰产对抗」是互联网安全黑话，PRD / CLAUDE.md 要求使用「异常行为检测」「业务违规行为治理」等通用工程表达。
- 「甲乙方」对非安全背景的 HR 不友好。

## 四、Experience 问题（按公司）

### 4.1 闪捷信息科技有限公司

#### 公司 Summary 过长

当前：
> 作为核心设计开发角色， 参与公司API网关、API审计、AI网关的产品的设计、开发、项目推广、客户现场支持等工作。

**问题**：
- 句子过长，信息密度低。
- 「参与」一词弱化了核心开发者定位（PRD 第 11.1 章要求体现绝对核心开发角色）。
- 可改为一句话定位，例如：「负责 API 安全网关、AI 安全网关与数据安全引擎的核心设计与开发」。

#### 项目 1：AI 安全网关平台

**问题**：
- Highlights 以「主要功能」开头，偏向产品说明书。
- 「支持3种代理模式……」是功能列表，不是 What/Contribution/Impact。
- 缺少 QPS +138%、RT -63% 等核心量化结果（这些结果当前写在第二个项目里）。
- 仍包含「已持续稳定运行超 2 年」，但 facts.yaml 已确认该描述属于 nginx_merry，不适用于 Nexis。
- 项目名是否统一为「Nexis」需要确认（已确认可以）。

#### 项目 2：API 网关

**问题**：
- 项目名过于泛化「API 网关」，粗料中对应的是「nginx_merry」。
- 第一个 bullet 是功能介绍，长达 100+ 字，信息过载。
- 第二个 bullet 包含「从网络层到应用层再到审计层」的实现细节，偏向 How/Why。
- QPS +138%、RT -63% 写在该项目下，但粗料确认这是 Nexis 对比 nginx_merry 的结果。当前表述容易让人误以为是同一项目的重构前后对比。

#### 项目 3：r_data_service

**问题**：
- 第一个 bullet 过长，包含多 crate 架构、四种服务形态、三阶段流水线、80+ 敏感数据类型、20+ 脱敏算法等大量细节。
- 这是「产品说明书」式写法，违反 PRD 第 3 章 Design Philosophy 和第 9 章 Information Density。
- 性能提升「5x+」已确认，但「整体链路吞吐 1x+ 以上」价值不高，可删或合并。

#### 缺少 5 个配套能力

当前简历只列出 3 个闪捷项目。PRD 和粗料中还有 secagent、AI-dataset、AI 研究、flint-ocr、file-inspector。需要决定这些作为独立项目还是并入核心项目。

### 4.2 玩物得志

#### 公司 Summary 较好，但可更聚焦

当前：
> 主导研发高并发业务风控引擎与文本反垃圾系统，负责企业安全治理与等保合规建设。

**评价**：相对清晰，但「主导研发」可加强为「独立设计并研发」或「从 0 到 1 搭建」。

#### 项目 1：业务风控引擎

**问题**：
- Highlights 以「围绕注册、登录、支付、提现等核心业务链路」开头，偏向场景描述。
- 缺少核心量化数据（10w+ QPS、<50ms 延迟、85%+ 命中率、年拦截 500w+），这些数据在粗料中已确认但未写入。
- 「黑灰产对抗」应改为「异常行为检测」或「业务风险」。

#### 项目 2：高性能文本反垃圾系统

**评价**：相对较好，但「多级缓存与规则预编译」是 How，可弱化。

#### 项目 3：企业级安全治理与等保合规

**问题**：
- 缺少量化 Impact（等保测评时间、漏洞收敛数量、安全事件下降率等）。
- 这是 Resume A 的重要项目，但在当前简历中显得薄弱。

### 4.3 蘑菇街

#### 公司 Summary 过长且含口语化表达

当前：
> 作为核心人员，主要从事 IM 系统，网络安全，HIDS，WAF 等开发维护工作；见证并参与了自研 WAF从 0-1，以及整个 mogu 安全逐步建成一个完备的安全体系的过程；在安全部负责和参与的项目较多，摘要如下

**问题**：
- 「见证并参与了」「mogu」「摘要如下」等表达口语化、不专业。
- 句子过长，超过 100 字。
- 没有给出一句话定位。

#### 项目 1：自研 IM

**问题**：
- 第一个 bullet 包含「分包组包、状态机、自动重连、加密与网络诊断」等实现细节，偏向 How。
- 缺少「亿级消息峰值」「99.9% 可用性」等已确认量化结果。

#### 项目 2：WAF 网关

**评价**：这是当前简历中写得最好的项目之一，但仍有优化空间：
- 「waf-gate + waf-server 进程分离架构」是 How，可简化为「设计 WAF 网关」。
- QPS 20K+、超时率 10%→0.01% 是核心数据，应更突出。
- 这些数据仅在蘑菇街项目使用，符合 CLAUDE.md 规则。

#### 缺少信息安全治理项目的影响力

「企业级信息安全治理」项目在当前 resume.yaml 中完全没有出现，但粗料和 facts.yaml 中有相关素材。

### 4.4 千方集团 / 中交兴路

#### 两段经历仍拆分为两个项目

PRD / facts.yaml 已确认两段经历合并为一个项目「CADN 车联网分布式接入与消息分发平台」。当前 resume.yaml 仍拆为「高并发接入模块」和「CADN 核心服务研发」。

#### 篇幅超过 P3 权重

作为早期经历（P3），当前两个项目共有 5 个 bullet，细节过多。应压缩为 2–3 个 bullet，突出规模和专利/奖项。

### 4.5 华为技术有限公司

#### 篇幅基本合适，但 Impact 可更聚焦

当前：
> 负责虚拟文件管理（VFM）模块研发，保障文件系统在电信级严苛环境下的高稳定性。阅读大量 Linux 内核与 C 源码，多次独立定位并解决内存泄漏、死锁、并发竞争等高级故障，受到团队认可。在此过程中建立的高可靠软件工程理念，持续影响后续系统设计与编码实践。

**问题**：
- 作为 P3 经历，这一段仍然偏长。
- 「受到团队认可」是主观描述，可删。
- 核心信息是：电信级工程纪律奠基。

## 五、全局 IA 问题

### 5.1 没有体现 Career Arc

PRD 第 4 章职业主线：
> Reliable Software Engineering → High Performance Infrastructure → Large Scale Backend → Enterprise Platform → Enterprise Security → AI Infrastructure

当前简历没有让招聘经理感受到这条主线。公司之间是并列关系，而不是演进关系。

### 5.2 没有体现 Hiring Risk Reduction

PRD 第 2 章要求每句话降低招聘风险。当前简历更多是在罗列能力和项目，没有主动回应招聘经理的顾虑：
- 会不会落地？
- 能不能维护？
- 能不能带团队？
- 是不是只会写 Demo？

### 5.3 信息预算超支

PRD 第 18 章要求 2 页 / 900–1200 词。当前简历：
- 内容量大，部分项目 bullet 过长。
- 早期经历（P3）细节过多。
- Core Skills 中包含低价值内容。

预估当前简历如果直接排版，信息量会超过 2 页舒适区。

### 5.4 Narrative Consistency 不足

PRD 第 19 章要求每家公司推进 Narrative。当前简历：
- Resume A 想表达 Security，但 Backend 项目（IM、CADN）没有从安全角度重构。
- Resume B 想表达 Infrastructure，但安全项目（风控、WAF）没有从基础设施角度重构。
- 同一段经历在不同 Branch 中应该有不同的强调点，当前没有体现。

## 七、建议的修改方向

### 7.1 结构层面

1. **拆分 Resume Branch**：不再维护单一 `resume.yaml`，而是创建 `data/resume-security.yaml` 和 `data/resume-backend.yaml`。
2. **重建 Summary**：每份简历的 Summary 前两句必须直接建立专业身份。
3. **重写 Core Skills**：按 Resume Branch 分别设计，删除低价值内容。
4. **压缩早期经历**：千方/中交兴路合并为 1 个项目，2–3 个 bullet；华为 1–2 个 bullet。
5. **增加蘑菇街安全治理项目**：Resume A 需要这个项目。

### 7.2 内容层面

1. **删除软技能和性格描述**：如「热爱技术、积极乐观」。
2. **删除实现细节和 Why/How**：如「为什么选 Pingora」「模块如何拆分」。
3. **删除产品说明书式描述**：如「主要功能包括……」。
4. **突出量化 Impact**：将粗料中已确认的数据写入对应项目。
5. **统一技术黑话**：如「黑灰产对抗」→「异常行为检测」。
6. **修复事实错误**：如 Nexis 的「稳定运行超 2 年」。

### 7.3 叙事层面

1. **每家公司增加一句话定位**，说明这家公司如何推进职业主线。
2. **每个项目按 Resume Branch 调整强调点**：
   - Resume A：安全能力、风险降低、合规、检测防护。
   - Resume B：性能、并发、架构、可靠性、工程化。
3. **确保 Career Arc 可见**：让招聘经理读完简历后，能感受到从工程纪律到 AI Infrastructure 的演进。

## 九、结论

当前 `data/resume.yaml` 是一份「项目堆砌型」简历，存在以下核心问题：

1. **定位模糊**：没有区分 Resume A/B，Summary 开篇是软技能。
2. **信息过载**：项目描述包含大量 How/Why/功能列表，违反 Information Density 原则。
3. **事实错误/缺失**：Nexis 稳定性描述错误、早期经历拆分、多个项目缺少已确认量化数据。
4. **叙事断裂**：公司之间没有形成 Career Arc，每家公司没有推进 Narrative。
5. **预算超支**：早期经历和项目细节过多，预计超过 2 页舒适区。

建议按 PRD v3.0 进行彻底重构，而不是局部优化。
